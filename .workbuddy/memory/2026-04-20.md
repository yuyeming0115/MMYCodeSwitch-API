## Phase 2.5：连锁崩溃 Bug 修复（关键！）

### 根因
**ProjectList.vue 渲染崩溃导致整个 App 瘫痪**：
1. `store.activeProjects` 中存在 `project_path` 为 undefined 的脏数据（可能是之前某次不完整写入）
2. ProjectList.vue 的 `truncatePath(undefined.length)` 抛 TypeError
3. Vue 错误边界传播 → 整个 AppContent 组件树损坏
4. **连锁后果**：底部按钮全部失灵、右键菜单不弹、添加供应商无反应、ConfigProvider theme 报错

### 修复内容
1. **normalizePath() 加空值防御** — 参数类型改为 `string | undefined | null`，返回空字符串而非崩溃
2. **doSwitch 遍历防御** — find 回调中增加 `proj.project_path &&` 前置条件，跳过脏数据
3. **catch 块安全处理** — 用 `e instanceof Error ? e.message : String(e)` 替代 `(e as Error).message`
4. **ProjectList.vue 全面防护**：
   - `truncatePath()` 参数加 undefined/null 防御
   - template 中所有字段访问加 fallback（`proj.name || '未知项目'`）
   - handleRemove 中的 name 引用也加了 fallback

### 经验教训
- **子组件渲染崩溃会向上传播破坏父组件的事件系统**
- **所有外部数据访问都必须做 null/undefined 防御**，尤其是从 Rust 后端反序列化来的数据
- console 日志是救命稻草——这次靠 `[doSwitch]` 日志定位到了具体崩溃行号 工作日志
