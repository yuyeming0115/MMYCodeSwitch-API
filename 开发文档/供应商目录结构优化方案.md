# 供应商目录结构优化方案

> 日期: 2026-05-12

## 1. 背景

当前 `.mmycs` 目录采用扁平结构，所有项目的会话数据混在 `projects/` 目录下（以哈希名命名），通过 `binding.json` 中的 `provider_id` 关联供应商。供应商信息单独存放在 `providers/` 目录下的 JSON 文件中。

对比 JCode 的做法——以供应商名作为顶层目录（如 `.jcode/2api/`、`.jcode/OpenRouter/`、`.jcode/ali/`），各项目、会话、插件等数据天然隔离在供应商目录内——归属关系一目了然。

## 3. 目标结构

### 3.1 目录结构

```
.mmycs/
├── config.json                   # 全局配置（精简，只保留全局设置）
├── window_state.json
├── .key
├── providers/                    # ← 按供应商分目录
│   ├── aliyun-glm5/              # 供应商目录（名称 = 规范化供应商名）
│   │   ├── provider.json         # 供应商配置（原 provider_*.json）
│   │   ├── projects/             # 该供应商下的项目
```

### 3.2 目录命名规则

供应商目录名从供应商名规范化生成：
- 取供应商 name 字段
- 去除特殊字符，中文用拼音或英文替代（阿里云百炼 → aliyun）
- 加模型名后缀区分（glm5、qwen36）
- 全小写，连字符连接

例如：
- `阿里云百炼-glm5` → `aliyun-glm5`
- `阿里云百炼-Qwen3.6` → `aliyun-qwen36`
- `foxcode` → `foxcode`
- `how88` → `how88`

项目目录名使用 `config.json` 中 `activeProjects` 的 `name` 字段，去除特殊字符。

### 3.3 config.json 变更

```json
{
  "language": "zh",
  "instances": [
    {
      "id": "default",
      "name": "默认实例",
      "active_provider": "aliyun-qwen36"
    }
  ],
  "activeProjects": [
    {
      "id": "proj_1776821599506",
      "name": "MMYCodeSwitch-API",
      "project_path": "D:/GitWork/MMYCodeSwitch-API",
      "provider": "aliyun-qwen36",
      "config_dir": "providers/aliyun-qwen36/projects/MMYCodeSwitch-API",
      ...
    }
  ],
  "backupExportPath": "D:\\GitWork\\MMYCodeSwitch-API\\backups",
  "version": 2
}
```

主要变更：
- `activeProjects[].provider_id` → `activeProjects[].provider`（改为目录名）
- `activeProjects[].config_dir` 更新为新的相对路径
- `instances[].active_provider_id` → `instances[].active_provider`

### 3.4 provider.json 变更

原 `provider_*.json` 内容基本保留，增加：
- `directory_name` 字段：规范化目录名
- 移除冗余的 `id` 字段（目录名即标识）

## 5. 需要修改的代码文件

### 5.1 Rust 后端 (src-tauri/)

| 文件 | 修改内容 |
|------|----------|
| `src-tauri/src/config.rs` | config.json 数据模型：`provider_id` → `provider`，`config_dir` 路径解析 |
| `src-tauri/src/provider.rs` | 供应商加载逻辑：从 `providers/<dir>/provider.json` 读取，目录扫描代替文件扫描 |
| `src-tauri/src/project.rs` | 项目加载逻辑：从 `providers/<provider>/projects/<name>/` 读取，创建时写入对应供应商目录 |
| `src-tauri/src/session.rs` | 会话路径解析：跟随项目 `config_dir` 变更 |
| `src-tauri/src/backup.rs` | 备份路径适配新结构 |
| `src-tauri/src/template.rs` | 模板路径适配 |

### 5.2 Vue 前端 (src/)

| 文件 | 修改内容 |
|------|----------|
| `src/types.ts`（或相关类型定义） | Provider/Project 接口字段名更新 |
| `src/stores/` | 状态管理中供应商/项目关联逻辑更新 |
| `src/components/` | 如有直接引用路径的组件，更新路径格式 |

## 6. 数据迁移步骤

1. **备份**：执行一次完整备份（现有机制）
2. **创建目录结构**：按目标结构创建供应商目录
3. **迁移供应商配置**：将 `provider_*.json` 复制并重命名为 `provider.json`
4. **迁移项目数据**：按映射关系移动项目目录
5. **更新 config.json**：重写配置文件中的引用
6. **验证**：启动应用检查所有项目正常加载
7. **清理**：删除旧的 `projects/` 目录和 `providers/` 下的旧文件

## 7. 兼容性考虑

### 7.1 数据版本号

在 `config.json` 中增加 `"version": 2` 字段标识数据结构版本，应用启动时：
- 版本 1：执行迁移逻辑
- 版本 2：直接加载

### 7.2 回滚方案

迁移前自动备份完整 `.mmycs` 目录到 `backups/`，如需回滚：
1. 恢复备份
2. 将 `config.json` 的 version 改回 1

## 8. 收益

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 项目归属识别 | 需要打开 binding.json 查看 | 目录路径即可判断 |
| 供应商数据隔离 | 无 | 天然隔离 |
| 导出/导入 | 需按 provider_id 筛选 | 复制供应商目录即可 |
| 磁盘浏览 | `ls projects/` 全是哈希名 | 清晰的树状结构 |
| 与 JCode 一致性 | 不一致 | 结构对齐，降低认知负担 |
